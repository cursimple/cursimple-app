# 课表时间栏、日志与提醒稳定性设计

日期：2026-05-05
状态：已按用户需求推进

## 目标

- 课表左侧节次时间拆成两行：第一行开始时间，第二行结束时间。
- 左侧节次栏尽量收窄，并在不同屏宽下自适应。
- 日志导出不限制 logcat 行数，并补充 App 自有结构化诊断日志。
- 日志自动清理按 3 天周期执行。
- 根据 Android 官方文档处理系统闹钟和精确闹钟权限差异，避免设置失败导致崩溃。

## 方案对比

推荐方案：保留现有界面与提醒架构，做定点加固。

- 优点：改动集中，不改变课表数据模型和提醒规则模型。
- 优点：系统闹钟失败、应用内提醒权限不足、通知权限不足都会返回明确结果或记录日志，不再向上抛出导致崩溃。
- 缺点：如果用户没有授予系统要求的精确闹钟权限，应用只能保存规则并提示部分提醒未完成设置。

备选方案一：彻底改成 WorkManager 提醒。

- 优点：不依赖精确闹钟特殊权限。
- 缺点：课程提醒时间不够精确，不适合作为课程闹铃主路径。

备选方案二：强制只使用系统时钟 App。

- 优点：避免应用自己持有闹钟调度。
- 缺点：不同厂商时钟 App 对 `ACTION_SET_ALARM` 支持不一致，失败时无法保证提醒规则可用。

## 设计边界

- 不把“临时兜底”作为正式架构手段。
- 不新增一套与现有提醒规则并行的数据模型。
- 不清理系统 `logcat`，只处理 App 自己生成的导出文件和诊断日志。
- 写入文档后不执行 git commit，提交由用户明确发起。

## 界面

日视图和周视图左侧时间栏都使用同一时间文本规则：

- 空时间显示为空。
- 有时间时显示两行：`startTime` 与 `endTime`。
- 不再显示 `08:00~09:40` 或 `08:00 - 09:40` 这种单行范围。

周视图的时间列从固定 60dp 改为按屏宽选择 44dp、48dp 或 52dp。日视图左侧节次栏从固定 62dp 改为 `widthIn`，让内容决定实际宽度，并保持最大宽度约束。

## 日志

新增 App 诊断日志：

- `cache/app-logs/app-diagnostics.log`：记录 App 生命周期、日志导出、日志清理、提醒调度汇总等结构化事件。
- `cache/plugin-logs/plugin-diagnostics.log`：保留现有插件诊断日志。
- 导出文件仍生成到 `cache/logs/`。

导出日志时：

- `logcat -d --pid=<pid> -v time` 不再传入 `-t`，不限制导出条数。
- 导出文件包含设备信息、App 诊断日志、插件诊断日志和当前进程 logcat。
- 文件 sink 继续做大小上限裁剪，防止 App 私有缓存无限增长。

自动清理：

- 使用 WorkManager 注册唯一周期任务，间隔 3 天。
- 清理 3 天前的导出文件与长期未更新的诊断日志。
- 手动“清空日志”仍立即清空 App 自有日志缓存。

## 闹钟稳定性

网上核对的 Android 官方要点：

- Android 12+ 精确闹钟可能需要 `SCHEDULE_EXACT_ALARM` 特殊权限，并应在调用前检查 `AlarmManager.canScheduleExactAlarms()`。
- Android 14+ 新安装 App 的精确闹钟权限可能默认被拒绝。
- `AlarmClock.ACTION_SET_ALARM` 需要通过外部时钟 App 处理，必须捕获没有可处理 Activity、权限或厂商实现异常的情况。

改造后：

- Manifest 声明 `SCHEDULE_EXACT_ALARM`。
- 应用内闹钟通道调用 `setAlarmClock` 前先检查权限；无权限时返回失败结果，不抛异常。
- 系统时钟通道不再依赖 `resolveActivity` 的结果作为唯一判断，直接尝试启动并捕获 `ActivityNotFoundException`、`SecurityException` 和其它运行时异常。
- 通知发出前检查通知权限状态；通知被禁用时只记录日志并结束。
- 创建提醒规则后汇总调度结果，失败时在状态消息中提示用户。

- 跑 `:core-reminder:assembleDebug` 与 `:app:assembleDebug` 验证提醒模块与应用编译。

- 手动检查课表日视图和周视图左侧时间为两行显示。
